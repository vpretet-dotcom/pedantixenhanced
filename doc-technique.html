01 — Présentation

Qu'est-ce que Pedantix Enhanced ?

Un jeu web de devinettes où le joueur doit retrouver le titre d'une page Wikipédia en proposant des mots un par un. Les mots trouvés sont révélés dans le texte, les autres colorent les blocs selon leur proximité sémantique.

🎮

Solo, Catégories, Quotidien, Surprise, Multijoueur

  • Mode quotidien synchronisé (minuit Paris)
  • Coopératif temps réel via Firebase

NLP Français embarqué

  • Normalisation Unicode (NFD + strip accents)

03 — Algorithmes NLP

Traitement du Langage Naturel

Le cœur algorithmique du jeu : comment déterminer si un mot proposé correspond à un mot du texte, et à quel point il en est proche.

🔤 Pipeline de normalisation

Chaque mot passe par 4 transformations pour maximiser les chances de correspondance :

mot brut
→
toLowerCase()
→
NFD decompose
→
strip accents
→
normalisé
// normalize("Géographie") → "geographie"
export function normalize(t) {
  return t.toLowerCase()
    .normalize('NFD')
    .replace(/[\u0300-\u036f]/g, '')
    .replace(/[^a-z]/g, '');
}

📝 Lemmatiseur français

15 règles morphologiques appliquées séquentiellement pour réduire les mots à leur forme de base :

RègleExemple
/(aient|ussent|eraient)$/marchaient → march
/(ais|ait|ons|ez)$/parlons → parl
/(er|ir|re)$/manger → mang
/(aux)$/ → 'al'journaux → journal
/(eaux)$/ → 'eau'gâteaux → gâteau
/(euses|euse)$/ → 'eux'joyeuse → joyeux
/(s)$/chats → chat
Contrainte : Le résultat doit faire ≥ 3 caractères et être différent du mot d'entrée, sinon la règle est ignorée.

📐 Word Vectors par n-grammes de caractères

Pour mesurer la proximité sémantique entre le mot proposé et les mots cachés, on génère un vecteur de dimension 512 basé sur les n-grammes de caractères (unigrammes, bigrammes, trigrammes) :

const V = 512; // Dimension du vecteur

export function wordVec(word) {
  const w = normalize(word), v = new Float32Array(V);

  // Hash function: Knuth multiplicative hash
  const h = (s, seed) => {
    let h = seed * 2654435761;
    for (let i = 0; i < s.length; i++) {
      h = Math.imul(h ^ s.charCodeAt(i), 2654435761);
      h ^= h >>> 16;
    }
    return Math.abs(h) % V;
  };

  // Unigrammes: chaque caractère
  for (const c of w) v[h(c, 0)]++;
  // Bigrammes: paires de caractères consécutifs
  for (let i = 0; i < w.length - 1; i++) v[h(w[i] + w[i+1], 1)]++;
  // Trigrammes: triplets de caractères
  for (let i = 0; i < w.length - 2; i++) v[h(w[i]+w[i+1]+w[i+2], 2)]++;

  return v;
}
Pourquoi des n-grammes de caractères ?
Contrairement aux word embeddings classiques (Word2Vec, GloVe) qui nécessitent des modèles de plusieurs Go, cette approche est 100% côté client, instantanée, et capture les similarités morphologiques : « national » et « nation » partagent les trigrammes « nat », « ati », « tio », « ion ».
Similarité cosinus
La proximité entre deux mots est calculée par cos(θ) = (A·B) / (‖A‖ × ‖B‖). Un score de 1.0 = identique, 0.0 = aucun rapport. Le résultat est affiché sous forme de feedback chaud/froid (🔥 > 0.75, ❄️ < 0.15).
04 — Game Loop

Boucle de jeu principale

De la sélection d'un article à la victoire, voici le cycle de vie complet d'une partie.

1. Récupération d'article

Trois stratégies selon la difficulté :

DifficultéSourcePool
FacileArticles les + vusTop 300
NormalArticles populairesTop 3000
DifficileArticles aléatoires∞
Filtres de qualité : Les articles < 800 caractères, les pages d'homonymie, les listes, et les articles avec < 80 mots significatifs sont automatiquement rejetés.

2. Tokenisation

Le texte brut est découpé en tokens, chacun enrichi de métadonnées :

// Chaque token contient :
{
  id: 42,           // ID unique
  text: "Géographie", // texte original
  lower: "géographie",
  norm: "geographie",  // sans accents
  lemma: "geograph",  // lemmatisé
  lemNorm: "geograph",
  vec: Float32Array(512), // vecteur
  isWord: true,
  isStop: false,      // mot vide ?
  revealed: false,    // trouvé ?
  bestProx: 0,       // meilleure similarité
  bestGuess: null    // mot le + proche
}

3. Algorithme de matching (guess)

Quand le joueur propose un mot, 6 comparaisons sont effectuées contre chaque mot caché :

// Pour chaque mot non-révélé du texte :
const match =
  lo === t.lower    || // 1. Correspondance exacte (minuscule)
  nm === t.norm     || // 2. Formes normalisées identiques
  lm === t.lemma    || // 3. Lemmes identiques
  lmn === t.lemNorm || // 4. Lemmes normalisés identiques
  nm === t.lemNorm  || // 5. Normalisé vs lemme normalisé
  lmn === t.norm;      // 6. Lemme normalisé vs normalisé

// Si aucun match → calcul de similarité cosinus
const sim = cosSim(guessVec, t.vec);
// Met à jour la meilleure proximité du bloc
Exemple concret : Le joueur tape « mangent ». Le lemmatiseur donne « mang ». Le texte contient « manger » (lemme: « mang »). Match via la règle 3 → le mot est révélé. ✅
05 — Multijoueur

Système coopératif temps réel

Architecture du mode multijoueur basé sur Firebase Realtime Database, avec synchronisation en temps réel des guesses et des mots révélés.

📡 Structure Firebase

sessions/
  {CODE_6_CHARS}/
    ├── host: "pseudo"
    ├── status: "waiting" | "playing" | "won"
    ├── createdAt: 1720100000000
    ├── startedAt: 1720100030000
    ├── players/
    │   ├── p_abc123/
    │   │   ├── name: "Alice"
    │   │   ├── color: "#6366f1"
    │   │   └── online: true  ← onDisconnect().remove()
    │   └── p_def456/ { ... }
    ├── article/
    │   ├── title: "Tour Eiffel"
    │   ├── titleToksJSON: "[...]"  ← tokens sérialisés (sans vecs)
    │   ├── secsJSON: "[...]"
    │   ├── totalWords: 342
    │   └── url: "https://fr.wikipedia.org/..."
    ├── guesses/
    │   └── {pushId}/ { word, found, sim, playerName, playerColor, timestamp }
    ├── revealed/
    │   ├── tour: true
    │   └── eiffel: true
    └── config/
        ├── difficulty: "easy"
        └── categories: ["geo"]

🔌 Flux Host (créateur)

  1. Génère un code à 6 caractères (A-Z, 2-9)
  2. Crée la session dans Firebase
  3. Écoute les joueurs qui rejoignent
  4. Lance la partie → fetch article + tokenize
  5. Sérialise les tokens (sans vecteurs) → Firebase
  6. Met le status à « playing »

🔗 Flux Guest (invité)

  1. Saisit le code à 6 caractères
  2. Rejoint la session (ajout dans players/)
  3. Écoute le changement de status → « playing »
  4. Télécharge les tokens depuis Firebase
  5. Reconstruit les word vectors localement
  6. Synchronise les guesses en temps réel
Astuce d'optimisation : Les word vectors (Float32Array de 512 dimensions) ne sont pas envoyés à Firebase — ils sont recalculés localement par chaque client. Cela réduit la taille du payload de ~95%.
06 — Mode Quotidien

Article du jour

Un article identique pour tous les joueurs chaque jour, avec leaderboard dédié.

🎲 Sélection déterministe par seed

L'article du jour est choisi via un hash déterministe de la date. Tous les joueurs obtiennent le même article sans communication serveur :

function dateSeed(dateKey) {  // dateKey = "2026-07-04"
  let h = 0;
  for (let i = 0; i < dateKey.length; i++) {
    h = Math.imul(h ^ dateKey.charCodeAt(i), 2654435761);
    h ^= h >>> 16;
  }
  return Math.abs(h);  // Nombre déterministe
}

// seed % pool.length → index de l'article dans le top 500
Fallback Firebase : Si Firebase est configuré, l'article est stocké en base pour garantir que tous les joueurs ont exactement le même, même si le top 500 de Wikipedia change en cours de journée.

⚡ Mise à jour incrémentale

Plutôt que de re-renderer tout le DOM à chaque guess, on utilise un diff ciblé :

  1. Indexer tous les .word-block par dataset.id
  2. Pour chaque token modifié, mettre à jour uniquement son élément
  3. Si révélé : remplacer le nœud (clone + classe flash-new)
  4. Si non-révélé : mettre à jour backgroundColor
  5. Scroll vers le premier mot nouvellement révélé
Complexité : O(n) par guess, où n = nombre total de tokens. Pas de re-render complet → 0 flash/flicker visible.
08 — State Management

Gestion de l'état

Un store centralisé inspiré de Vuex/Redux, mais sans framework. Toutes les variables partagées sont dans state.js, avec des setters explicites.

🗃️ Groupes d'état

🎮 Game State

  • game — données de la partie
  • gameStartedAt — timestamp
  • currentDiff — easy/normal/hard
  • scorePublished — flag
  • tid — compteur de tokens

👥 Colab State

  • colabMode — solo/host/guest
  • colabSessionCode
  • colabPlayerName
  • colabPlayers — objet joueurs
  • colabUnsubscribers — cleanup

📅 Daily State

  • dailyActiveDate
  • dailySelectedDay
  • dailySort
  • dailyCountdownInterval
Pourquoi des setters ? Les ES Modules exportent des bindings en lecture seule. On ne peut pas écrire import { game } from './state.js'; game = newValue; — il faut passer par setGame(newValue). C'est une contrainte de la spec ES6, pas un choix de design.
09 — Firebase

Backend Firebase

Firebase Realtime Database est utilisé pour 3 fonctionnalités : multijoueur, leaderboard, et mode quotidien.

🔥 Chargement lazy

Firebase n'est chargé que quand c'est nécessaire — le joueur solo ne télécharge jamais le SDK Firebase :

async function getFirebase() {
  if (_fb) return _fb;
  // Dynamic import — chargé à la demande
  const [appMod, dbMod] = await Promise.all([
    import('https://...firebase-app.js'),
    import('https://...firebase-database.js')
  ]);
  // ~130 KB au lieu de 0 pour le solo
}
